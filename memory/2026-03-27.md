
## 2026-03-26 to 2026-03-27: Beam YAML & Event-Driven Dataflow Architecture PoC
- **Context:** Jason's team needed to ingest CSVs from GCS to BigQuery using Beam YAML. They initially tried a single Streaming Dataflow job listening to Pub/Sub, but it failed/OOM'd because defining >10 different schemas in one YAML caused DAG explosion and cross-language Expansion Service issues.
- **Explorations:** 
  - Created a parameterized Beam YAML batch job (`beam-yaml-csv-to-bq`).
  - Explored triggering Dataflow from another Dataflow using a custom Python Provider (`beam-yaml-job-b`, `beam-yaml-python-provider`). This proved to be an anti-pattern (Worker nodes missing dependencies during CloudPickle deserialization, requiring `setup.py` distribution, plus IAM over-permissioning).
  - Explored launching Beam YAML natively via Python using `apache_beam.yaml.yaml_transform`. (Java cannot natively parse Beam YAML).
- **Final Architecture (The "Elegant" Solution):** 
  - Designed an Event-Driven Serverless architecture: **GCS (Notification) -> Pub/Sub (Push Subscription) -> Cloud Run (FastAPI) -> Dataflow Batch Job (Beam YAML) -> BigQuery**.
  - Cloud Run acts as a micro-orchestrator, avoiding Dataflow streaming idle costs and DAG explosion.
  - Implemented the Pub/Sub to Cloud Run trigger using Terraform (`pubsub-cloudrun-poc` repo).
  - Implemented the Cloud Run service that dynamically compiles and submits Beam YAML batch jobs to Dataflow (`cloudrun-dataflow-trigger` repo).
- **Deliverables:**
  - Repos created for Jason (`nvd11` on GitHub): `beam-yaml-csv-to-bq`, `beam-yaml-job-b`, `beam-yaml-python-provider`, `pubsub-cloudrun-poc`, `cloudrun-dataflow-trigger`.
  - Authored a detailed technical blog post (CN/EN) documenting the architecture and the reasoning behind moving from Monolithic Streaming to Event-Driven Cloud Run + Batch.
  - Generated system architecture diagrams using Nano Banana API.

  - Wrote `infra/cloudbuild.yaml` to automate `terraform plan` and `apply`, running under the user's high-privileged service account.
  - Drafted the Dataflow pipeline using Beam YAML (`dataflow/pipeline_template.yaml`) with Jinja variables (`{{ input_csv_path }}`, `{{ target_bq_table }}`).
  - **Learning/Fix**: When using Beam Python SDK's `YamlTransform(yaml_string)`, the string cannot start with the `pipeline:` key (it throws a `Missing type specification at line 1` error). To satisfy the user's requirement to keep `pipeline:` in the raw YAML files (for company standards), I implemented a Python workaround using `yaml.safe_load()` to extract `parsed_yaml['pipeline']` before passing it to `YamlTransform`.
  - Tested manual end-to-end execution of the pipeline using a dynamically generated Dataflow job, which successfully loaded a test CSV from the Landing Bucket to BigQuery.
