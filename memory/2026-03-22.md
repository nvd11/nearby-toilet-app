
## 2026-03-22 - Automated RAG Evaluator Project Execution
- **Context**: Working with Boss Jason on his take-home assignment for the Director of AI Software Engineering role at Standard Chartered. The project builds a production-grade RAG evaluation framework using GCP, Cloud SQL (PostgreSQL + pgvector), and LangChain.
- **Phase 1 (Data Ingestion)**:
  - Validated that the `ingestion_runner.py` successfully processed the HSBC Annual Report (2544 chunks, fully embedded using Gemini API) and upserted into the Cloud SQL database.
  - Refactored `PgVectorDAO.upsert_document_transactionally` to use a single `cursor` object passed to helper methods (`_insert_document_record`, `_upsert_topics`, `_map_document_topics`, `_bulk_insert_chunks`). This enforces strict ACID transactions (all-or-nothing commits/rollbacks) rather than relying on high-level ORM sessions.
  - Fixed a logical bug in `_upsert_topics` where `ON CONFLICT DO UPDATE` was falsely counting as inserts. Leveraged PostgreSQL's implicit system column `xmax` (`RETURNING topic_id, (xmax = 0) AS is_inserted`) to accurately distinguish between true inserts and updates.
  - Enhanced the ingestion runner to output a highly professional, human-readable summary console log detailing soft-deletes and inserts per table.
- **Testing & Quality Assurance**:
  - Implemented comprehensive `pytest` suites for `PyMuPDFLoader`, `LangchainRecursiveChunker`, `GeminiEmbedder`, and `PgVectorRetrieverDAO`.
  - Employed advanced mocking techniques (e.g., mocking `async with conn.cursor()` and `google-genai` SDK calls) to ensure tests run fast without hitting real databases or APIs.
  - Resolved a VS Code Test Explorer discovery issue (`Test result not found`) caused by a rogue `test_emb.py` script failing on import due to a missing API key, and by fixing the `python.envFile` path in `.vscode/settings.json` to point to `.env.test`.
- **Phase 2 (Retrieval & Generation)**:
  - Designed the Retriever architecture focusing on SOLID principles. Reused the `GeminiEmbedder` but explicitly differentiated `task_type="RETRIEVAL_DOCUMENT"` (for chunks) and `task_type="RETRIEVAL_QUERY"` (for user queries) to implement **Asymmetric Retrieval**, significantly improving vector search accuracy.
  - Extracted hardcoded hyperparameters (e.g., `CHUNK_SIZE`, `EMBEDDING_DIMENSION`, `RETRIEVAL_TOP_K`, `RETRIEVAL_SIMILARITY_THRESHOLD`) into the centralized Pydantic `settings.py` and `.env` files.
  - Implemented `PgVectorRetrieverDAO` with native `psycopg3`. Features predicate pushdown for the similarity threshold (`(1 - (dc.embedding <=> %s::vector)) >= %s`) and dynamic metadata pre-filtering via SQL `JOIN`s, avoiding the N+1 query problem associated with ORMs.
  - Upgraded to `Gemini-2.5-pro` for the LLM judge/generator.
  - Complied with interview requirements by strictly implementing the generation phase (`LangchainGeminiGenerator`) using **LangChain Expression Language (LCEL)** (`prompt | llm | StrOutputParser()`). Updated `langchain-google-genai` dependency to `3.2.0` to avoid pulling in the heavy, legacy `google-api-python-client`.
- **Database Schema Design Iterations (Crucial Feedback from Jason)**:
  - Identified a critical missing piece for the automated evaluator: a table to log the actual Q&A interactions (`Question`, `Context`, `Answer`).
  - Initially proposed adding `evaluation_queries` with a `run_id` foreign key. Jason astutely pointed out this violates database normalization (3NF) and mixes operational logs with evaluation tracking.
  - **Refactored Schema**: 
    1. Created a clean `query_history` table to log all traffic (real users + automated tests). Removed `session_id` in favor of a universal `queried_by` audit field.
    2. Introduced a many-to-many bridging table `run_queries` to map specific evaluation runs (`evaluation_runs`) to a subset of queries in `query_history`.
    3. Jason identified that embedding `ground_truth` directly into the transactional `query_history` table is an architectural flaw (mixing static benchmark datasets with append-only operational logs). Pending removal/refactoring of `ground_truth` from the schema.
