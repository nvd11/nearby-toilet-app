# 资深架构师/中间件专家 面试题与参考答案

## 一、 容器化与混合云架构 (OpenShift / Kubernetes / Cloud Pak)

**1. 云原生迁移痛点：将传统架构迁移到 OpenShift/Cloud Pak，最大的技术痛点是什么？如何解决？**
*   **参考答案要点**：
    *   **痛点**：状态管理（Session/分布式锁依赖单点）、庞大的内存占用（传统JVM预热慢、内存消耗大）、文件系统依赖（本地日志、上传文件）、服务发现机制不同。
    *   **解决**：无状态化改造（Session外置到Redis）、调整JVM参数（使用容器感知的参数如 `-XX:+UseContainerSupport`，或迁移到OpenJ9/GraalVM）、存储挂载改造（使用PV/PVC或对象存储OSS）、引入K8s Service/Ingress替代传统的硬件负载均衡。

**2. 跨地域高可用：在混合云架构下，如何设计跨地域（Cross-Region）的高可用（HA）和灾备（DR）方案？**
*   **参考答案要点**：
    *   **网络层**：使用全局负载均衡（GSLB/DNS路由），基于健康检查实现流量切换。
    *   **应用层**：无状态部署，集群间不强同步。
    *   **数据层**：利用底层数据库的跨DC同步机制（如MySQL的半同步复制、云厂商的底层块存储异步复制）。在OpenShift层面，可利用 Red Hat Advanced Cluster Management (RHACM) 管理多集群策略。

**3. K8s底层排错：遇到K8s网络（SDN/CNI）和存储（PV/PVC）故障时的排查思路？**
*   **参考答案要点**：
    *   **网络**：检查跨节点Pod通信（`ping`, `traceroute`），查看CNI插件日志（如Calico/Flannel），检查 iptables/IPVS 规则是否正确下发，排查 CoreDNS 状态。
    *   **存储**：检查 PVC 的 Pending 状态原因（`kubectl describe pvc`），确认 StorageClass 供应者是否正常，登录宿主机检查挂载点目录权限和磁盘IO，排查 CSI 驱动组件日志。

**4. 平滑迁移策略：旧有的大型单体应用（依赖WebSphere集群特性）如何平滑迁移？**
*   **参考答案要点**：
    *   **绞杀者模式 (Strangler Fig)**：在网关层做流量分发，将新功能剥离为微服务在K8s运行，老系统逐步打包进胖容器（Fat Container）运行。
    *   利用 WebSphere Liberty 作为过渡容器运行环境，替代庞大的 ND 集群，去除对 WAS ND 部署管理器（Dmgr）的依赖。

**5. 容量规划：如何进行性能监控和容量规划？HPA指标制定？**
*   **参考答案要点**：
    *   使用 Prometheus + Grafana 进行监控。
    *   HPA 不仅依赖 CPU/Memory（基础指标），对于Java应用，更推荐引入自定义指标（Custom Metrics，如 QPS、HTTP请求延迟、线程池队列长度、甚至业务队列深度）。
    *   容量规划需结合压测（JMeter/LoadRunner），找出单Pod性能拐点。

## 二、 企业级中间件与极限调优 (WebSphere / WebLogic / JVM)

**6. JVM与中间件协同调优：GC与线程池、连接池调优如何协同？**
*   **参考答案要点**：
    *   木桶效应，需匹配。Web容器最大线程数决定了并发处理上限，数据库连接池大小决定了向下游的吞吐。
    *   如果连接池太小，线程池中的请求会阻塞等待获取连接，导致响应慢；如果线程池太大，会增加CPU上下文切换开销，并且产生大量对象加重GC负担。
    *   调优经验公式：`数据库连接池 = ((核心数 * 2) + 有效磁盘数)`。

**7. 复杂OOM排查：排查OOM或内存泄漏的完整步骤？**
*   **参考答案要点**：
    *   保留现场：配置 `-XX:+HeapDumpOnOutOfMemoryError`。
    *   分析 Dump：使用 MAT (Memory Analyzer Tool) 或 JProfiler，查看 Dominator Tree（支配树）和 Histogram。
    *   找 GCRoots：分析哪些大对象（如巨大的List、未关闭的流、ThreadLocal泄漏）被强引用导致无法回收。
    *   结合代码定位修复。

**8. 生产环境死锁：CPU 100% 或死锁时的第一步止血操作和排查命令？**
*   **参考答案要点**：
    *   **止血**：切断流量（通过网关摘除该节点），保留现场（不能立刻重启）。
    *   **排查**：
        1. `top -Hp <pid>` 找出CPU占用最高的线程ID。
        2. 将线程ID转为16进制（`printf "%x\n" <tid>`）。
        3. 使用 `jstack <pid>` 导出线程快照，搜索该16进制ID，看具体停在业务代码的哪一行，或者是否有 `BLOCKED`/`Found one Java-level deadlock` 提示。

**9. 类加载器地狱：处理过类冲突导致的系统故障吗？**
*   **参考答案要点**：
    *   **原因**：应用自带的jar包与WebSphere/WebLogic系统级别的jar包（如XML解析、SLF4J）冲突。
    *   **解决**：调整应用服务器的类加载策略为 `PARENT_LAST`（应用优先），或在 `weblogic.xml` 中配置 `<prefer-web-inf-classes>`。深入排查可使用 `-verbose:class` 打印类加载路径。

**10. 架构演进：向轻量级迁移时，架构层面需做出的妥协？**
*   **参考答案要点**：
    *   放弃应用服务器级别的集中式管理和Session集群复制，改用分布式缓存（Redis）。
    *   放弃 JTA/XA 这种强依赖中间件的分布式事务机制，改用业务层的最终一致性方案。
    *   监控从 JMX 转向拉取式的 Prometheus 暴露端点（Actuator）。

## 三、 分布式系统架构设计

**11. 服务拆分：如何利用DDD合理划分微服务边界？**
*   **参考答案要点**：
    *   通过事件风暴（Event Storming）梳理业务流程。
    *   识别限界上下文（Bounded Context），上下文内部要求强一致，上下文之间通过领域事件（Domain Event）异步解耦。
    *   避免拆得过细导致网络开销剧增和分布式事务泛滥，高内聚低耦合。

**12. 分布式事务：什么时候选2PC、TCC、SAGA或最终一致性？**
*   **参考答案要点**：
    *   **2PC (如Seata AT)**：适合内部系统，对一致性要求高，并发相对不极端的场景（有锁开销）。
    *   **TCC**：适合金融转账等需要强隔离性、有中间状态的业务，需要业务端编写Try/Confirm/Cancel逻辑，开发成本高。
    *   **可靠消息最终一致性**：最推荐的互联网架构方案，通过本地消息表或RocketMQ事务消息实现。

**13. 多级缓存架构：如何设计以及解决穿透/击穿/雪崩？**
*   **参考答案要点**：
    *   **穿透**：布隆过滤器（Bloom Filter）拦截，或缓存空值（设置短过期时间）。
    *   **击穿**：热点Key突然失效导致DB瞬时被打满。加互斥锁（如Redisson `tryLock`），只让一个请求去查DB并回写缓存。
    *   **雪崩**：大面积Key同时失效。给过期时间加上随机数（jitter）；确保Redis集群高可用。

**14. 高并发防御：限流、熔断和降级策略？**
*   **参考答案要点**：
    *   **网关层限流**：基于IP或UserID的令牌桶/漏桶算法限流。
    *   **应用层熔断**：当下游服务超时或错误率达到阈值（如Sentinel/Hystrix），自动切断调用，快速失败保护主系统。
    *   **降级**：非核心业务（如评价、推荐）降级为默认返回值，保障核心交易链路的可用性。

## 四、 CI/CD 与 DevOps 实践

**15. 流水线优化：如何优化复杂的CI/CD耗时？**
*   **参考答案要点**：
    *   并行执行不依赖的阶段（如单元测试与Sonar扫描并行）。
    *   利用缓存机制：挂载 Maven `.m2` 依赖目录、复用 Docker 镜像构建层（Layer Cache）。
    *   将大型单体拆分为独立部署的微服务，减小单次构建体积。

**16. 零停机发布：蓝绿/金丝雀与安全回滚？**
*   **参考答案要点**：
    *   **蓝绿**：全量部署新版本，通过切换 Ingress/Router 权重瞬间完成。回滚只需切回权重。
    *   **金丝雀**：按比例（如5%）引流到新版本，监控无异常后逐步放量。结合 Istio/Service Mesh 可实现精准的Header/User级流量控制。

## 五、 危机处理与领导力

**17. 救火案例：高压下定位并挽回项目（开放题）**
*   **参考答案要点**：考察系统性思维（先止血、后排查、再复盘），以及跨团队调配资源的能力。

**18. 技术与业务的博弈：处理不合理上线时间（开放题）**
*   **参考答案要点**：不直接说“不”，而是给出“带前提的选项”。例如：可以按期上线，但需要砍掉边缘需求/降级非核心功能；或者上线后安排技术专项迭代修复技术债。

**19. 技术生命力：如何保持敏锐度完成转型（开放题）**
*   **参考答案要点**：从解决实际业务痛点出发引入新技术，而不是为了技术而技术；建立团队内部分享机制；持续关注开源社区。

**20. 简历空白期核实（2008-2014）**
*   **参考答案要点**：诚实回答，无论是创业、深造、做自由职业还是因为个人原因休息，重点是强调在这期间**技术并没有落下**，或者这段经历锻炼了怎样的管理/沟通能力。
