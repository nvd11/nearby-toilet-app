# ADA 项目
好接下来我会介绍一下Ada这个项目

Ada的全称Ask Compliance Digital assitant 中文名就是Ask Compliance 数字助理


好接下来我会介绍一下项目背景Ask Compliance是一个汇丰risk & 部门的一个问答系统

他的user 主要是银行的 前台用户 可以让他们去载系统上 提出一些 合规方面的问题， 然后系统会根据用户的国家会在后台自动把问题分配给最近国家的合规专家 然后合规专家们必须去查阅一些相关的政策文档然后在系统上手动的回答这些问题

所以这个系统还是挺依赖于大量的合规专家方面的人力资源而用户们希望一些常见的问题能交 给ai来回答这样能大大减少合规专家方面的人工成本 

而Ada就是为了实现这个功能


整个Ada的项目其实分三个系统第一个就是ask compliance - AskC
第二个就是RC data platform - RCDP
第三个就是gen ai system  -GenAI
第4个是global policy & procedures system- gpps

其实这个系统也是一个典型的rag系统

他有三个主要的工作流 
第一个是document embedding, GPPS 会不断推送更新的policy /procedures document to RCDP （存于RCDP的BQ 和bucket）
如果RCDP detect 到DOC 和ADA相关， 则会吧documnet 数据推送到 GenAI 提供的embedding api, 这样GenAI 那边就会构建一个用于ADA的 vectorDB（alloydb in GenAI） 作为知识库

note： 这个flow 不会经过ochestration svc

第2个就是核心的问答工作流

RCDP 会build 一个ochestration svc

一旦用户在ASkC 提交一个问题form, AskC 如果判断这是一个可以用ai 回答的scope 内， 就会把querydata 发送到RCDP的一个pubsub 队列

Ochestration svc 会从队列里 提取query
然后会利用python 的相关库和bm25算法判断问题与用户选择主题的相关度和问题本身的数据敏感度还有问题的复杂度如果合适的话 就会 调用genAI 系统的RAG Agent query 接口

GenAI会参考vector知识库里的policy /procedure data返回ai rsponse和相关的指标， 例如参考的policy chunk的匹配度分数等 给ochestraction svc orchestration svc 再通过另一个pubsub队列传输回给AskC

这样AskC的合规专家就有了ai 的参考， 大大节省了合规专家的查阅文档的成本， 而且RcDP 配合genAI在无需侵入性修改下就为AskC带来了Ai的能力

第3个 工作流就是model evaluation/monitoring


AskC 会把被ADA处理过的query 和ai rsponse和user feedback发给RCDP

RCDP会把这些数据存下来（BQ）
再配合数据分析师设计一个evaluation api,  处理这些数据为每个ai rsponse 生成一些相关的model evaluation 指标， 存放于rcdp的另一个BQtable  然后 MI系统会获取evalution result数据生成相关报表， DATA analysic和genAI i系统的技术人员会基于报表的数据对 RAG agent 做持续改进。

##  Repid2 model1 alert triage projefct

简称 model1 project

背景
Rapid2 是我们Regulatory Compliance 部门中用于接收和初步分类处理 regualatory alert 的系统

之前， 需要大量人力资源去分析处理这些alert数据

我们集成了ML model 的能力for 这个工作流

这个项目共包含4个application

1. Rapid2
2. 就是RC data platform - RCDP    也是我所在的项目
3. compliance machine learning platform  - CMLP   Data Anaylsis（Compliance Analysis-CA） 会在cmlp train/develop ML-model
4. model execuion env 其实也是RCDP的一部分


第一条flow-数据准备和model dev
Rapid2 会将所有历史和更新的相关alert 和triage 数据发送到RCDP, by daily batch,   存放与RCDP BQ里的physical tables里 ， 然后RCDP 根据数据字典(主要是各表每个column的敏感度（public/internal/restircted/highly restrtic）） 和entitlement engine 创建for CMLP 下游系统的的BQ auth views

CA user 会通过cmlp（基于kubeflow, 其实就是gcp上的web jupyter notebook） 通过python BQ sdk 和SQL访问auth view的数据， 主要是alert 和 triage 的相关历史数据去做model training/development


第2条 flow- model deployment
CA user 是数据科学家， 他们虽然懂得如何 dev model ， 但他们毕竟不是专业的软件工程师， 所以他们不懂得如何productionize ， 所以我们RCDP own了一个model exccution env, 并构建了一个CICD pipline 自动部署user 从cmlp 开发的MOEL部署到MODEL exec env, 并执行e2e的测试， 最终部署生成和负责production monitoring

第3条flow 是model bisiness process flow
rapid2 会通过streaming etl 和 发送新的alerts 数据到RCDP 的pubsub topic, RCDP 通过streaming etl pipeline把数据处理到RCDP conformed layer 后， 会调用model exec env 的api 调用model处理alert 数据， 生成recommendation data 写回RCDP 的model output table.


第4tiaoflow是 recommendation reverse flow
RCDP 会 建立另一条streaming pipline 把recommendation record 发回给Rapid2， 这样Rapid2 的user就会收到哦啊model 的outcome数据

第5条flow 是model evaluation flow

Rapid2 在daily 把alert 和user 真正的triage decision数据发到RCDP ， 通过而evaluation model 去比较recommendation 和 user decision数据和其他相关的数据生成model的evalution 指标。

通过RCDP和cmlp的合作， 我们也再没有对rapid2 这个business app 侵入式修改的情况下， 为business app enable 了ai的能力
